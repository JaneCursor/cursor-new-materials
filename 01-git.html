Что же такое системы контроля версий (CVS)

– CVS — это система, сохраняющая изменения в одном или нескольких файлах так, чтобы потом можно было восстановить определённые старые версии

С помощью таких систем, многие разработчики могут сотрудничать друг с другом, не беспокоясь о проблемах или несоответствиях в файлах.

Все наработки по проекту всегда хранятся у нас, и мы в любой момент можем вернуться к старой версии или даже скомбинировать разные решения разных версий.

Простой пример CVS

Мы копируем файлы текущего проекта, указывая дату и время  создания копии, такой подход еще называют backup’ом, или более развитые – локальные системы контроля версий

Центральные системы контроля версий

В общем случае, делают то же что и локальные системы контроля версий, но на удаленном сервере.  Самый известный и до сих пор используемый пример ЦСУВ – Subversion(SVN)

Распределенные системы управления версиями

Примеры таких систем: git, mercurial, bazaar, darcs
Основное отличие таких систем в том, что клиенты, всегда забирают полную копию репозитория себе. Благодаря такому подходу, даже если сервер потеряет все данные, их можно моментально скопировать у кого-либо из клиентов и все снова заработает как нужно. Уничтожить данные из такой системы случайно практически невозможно.

Установка GIT

Для Windows: https://git-for-windows.github.io

Для MacOS: https://sourceforge.net/projects/git-osx-installer/

Мы рекомендуем устанавливать расширение git bash, как отдельную программу. Чтобы вы не путались с терминалом Windows.

Настройка Git

Обычно выполняется единоразово

  1. Проверяем работает ли git. Открываем git bash.
  2. Вводим в консоль команду git
  3. Теперь укажем наше имя и почту, которые будут  видны при коммите в репозиторий
  4. Проверим указаны ли наше имя и почта
        $ git config --global user.name "Your Name"
        $ git config --global user.email your@email.com
        $ git config —-list
        ИЛИ
        $ git config user.name

GitHub и BitBucket

Это хостинги для удаленных репозиториев Или иначе – место где можно разместить удаленный репозиторий

У обоих платформ есть подобие социальных сетей,  можем создавать команды и работать над проектами  с полной поддержкой pull request и issue

У каждой платформы есть свои плюсы, или минусы, но обе эти платформы работают с гитом, и нам абсолютно неважно где мы размещаем код

Команды, используемые в терминале

  •                 git add .  – добавляет файлы в коммит. Обычно используется «.» или «*» для добавления всех файлов.
                Чтобы добавить один файл – указывается его имя и расширение 
            
  •                 git commit -m "Сообщение описывающее коммит" – С помощью этой команды мы выполняем «коммит» изменений. То есть указываем что текущий слепок проекта нужно сохранить.
                Можно указать без аргумента -m, в таком случае откроется текстовый редактор по умолчанию, в котором нужно будет описать коммит.  Если это произошло случайно, введите :q и нажмите Enter
            
  •                 git push origin master – «Пушим» или отправляем в репозиторий наши слепки проекта.
                origin – название удаленного репозитория (обычно выбирают origin, но в теории может быть любое слово)
                master – название ветки в которую добавляем изменения (master – главная ветка в репозитории, может быть любой)
            
  •                 git init ./ – инициирует пустой репозиторий в текущей директории . Так же можно указать любой путь.
            
  •                 git remote add origin URL – Добавляем адрес удаленного репозитория, в который будем пушить изменения
               origin – название удаленного репозитория (обычно выбирают origin, но в теории может быть любое слово)
            
  •                 git pull origin master – команда для получения изменений из репозитория.
